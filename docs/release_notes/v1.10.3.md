# v1.10.3 — 本地音频运行时修复与 CI 改进

## Bug 修复

### 本地 TTS/ASR DLL 架构选择修复

`audio init` 从 Go module cache 复制 sherpa-onnx DLL 时，`copySherpaLib` 按目录遍历顺序取子目录，`i686`（32-bit）排在 `x86_64`（64-bit）前面，导致 Windows 上始终复制 32-bit 的 `sherpa-onnx-c-api.dll`，加载失败（`not a valid Win32 application`）。

- **`cmd/audio_init.go`**: `copySherpaLib` 分两轮扫描，优先匹配当前架构子目录（`amd64`→`x86_64`, `arm64`→`aarch64`），无匹配时 fallback 到任意目录
- **`cmd/audio_init.go`**: 新增 `releaseTag()`，dev build 版本号（如 `v1.10.2-12-gbcfd94c-dirty`）自动截断为基础 release tag，确保 `downloadAsset` 能访问到正确的 release URL
- **`internal/audio/sherpa_ffi.go`**: 依赖检查只查当前平台的文件，不再把 `.dylib`/`.so` 误报为缺失；加载失败时打印具体的 DLL 路径

### CI Release Asset 补全

`ensureRuntimeLibs` 的第三步依赖从 GitHub Release 下载 standalone asset，但 `sherpa-onnx-c-api.dll` / `libsherpa-onnx-c-api.*` 等运行时库在 CI 构建后被删除或未复制到上传目录，导致 standalone asset 缺失。

- **`.github/workflows/ci.yml`**: Windows 构建后只删 exe，保留 DLL；Unix 构建后将 `libsherpa-onnx-c-api.*` 从 module cache 复制到当前目录供上传
